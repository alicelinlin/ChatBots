Week 1 (April 17th-april 23rd)

Week 1 is the planning stage. To make a prototype, of something as complex as a chatbot can be, we must first investigate to ascertain what may be possible and realistic in this project. And then start to investigate how we could proceed to the building and testing phase.

After researching(‘googling’) some chatbots that are already out there. And looking for potential services that could assist in creating something marketable.

The research brought two likely candidates for a prototype. The first being Microsoft Azure (they offer a free trial for testing and development), and Discord (free API calls up to 1,000 per day). Both are claiming to have an easy to build chat bot framework. So, the investigation led first with these two.

Week 2 (April 24th- April 30th)

After some more research, Discord proved to be too limited for the uses of this project. The bot framework they support is to easily create a bot within a discord server. The framework they created is a coding package for use in NodeJS and Python which makes it easy to create some basic bot logic then send it directly to Discord server where it will be rendered in your Discord channel. So, in other terms it is a bot that can only be used while using discord. This does not suit us as we require a bot that can be exported and used outside of this environment. With Microsoft Azure left we continued to investigate this as a possibility.

Week 3 (May 1st – May 7th)

Exploring Microsoft Azure

After some more investigation Microsoft Azure offers something called a cognitive service. And this can be used to create “chatty” QnA Bot. Through reading the documentation of this service, I began to create a QnA Bot.

This process was tedious. The Microsoft Azure platform is neither intuitive nor user friendly. But with the help of the supporting documentation, I could create a “Knowledge Base”. This forms the basis of the QnA function. The platform will accept FAQ URLs from websites. So, you simply pass the URL into the field where it is required, and their system reads the webpage to procure the Questions and answers. After feeding the knowledge base some FAQ pages on Covid-19. You “publish” the knowledge base then Azure has provided a chat window to test and interact with the bot. Success! It works.

But this chat window exists only the Microsoft Azure platform. The question then becomes how to produce a product from this. But the answer comes quite simply in the form of an API supplied by Microsoft. At this point it becomes clear that in order to create a running prototype we will need to use code to create one using the API that Microsoft provide. I have some experience in developing programs with JavaScript. So, I begin working on the App.

The idea originally was to create a front facing application for the chat bot using the Microsoft API to interact with the knowledge base to get answers from the users’ questions, then render this on a chat window using JavaScript. The Microsoft API requires a key for authentication, which I assumed could be hidden using the GitHub environment secrets so we could incorporate the chat bot on the GitHub pages assignment website.

But after completing a functioning prototype this way, I discovered that these environment variables do not function as I assumed. These environment variables are for use in the GitHub Actions API and are not for use GitHub Pages Applications. In fact, all GitHub pages does is serve static HTML and all scripts and style sheets linked to it. So, there would be no way to keep my API key safe from prying eyes while using only a front facing application. What I would have to do is program a server to do the API call and leave the front end the job of dynamically rendering the HTML questions and answers from our user and the Bot response.

Node JS

>Week 4 (May 8th – May 14th)

During Week 3 I began to create the Server using NodeJS with Express as a framework. The process is quite simple using Express. The process was quite swift. To call the Microsoft API I used AXIOS. This is a promise-based API library that makes API calls simpler. After creating the server, a new workflow is created. The user will type a message using the chat window, hit Send, the front-end JavaScript then renders the question on the screen and passes the question to the back end using AXIOS. Once received by the Backend the Microsoft API is called, again using AXIOS. Using the, now secured, API key and the response is delivered back to the front end for rendering in the Chat Window. If an error occurs this is caught but no message is delivered to the user informing them of this. This is a feature that could be added in future development.

Once this process is completed, I use the time to style window and add the chat bubbles using CSS. And make the application appear more professional and pleasing to the eye. But from the next issue is lurking. Because after all this work I have nowhere to host it so it can be seen by users, investors, or assessment graders. From here I decide to host it using Heroku. Its free, easy to use and fits the purposes of this assessment.

The process of having a project hosted by Heroku is normally easy. But in this case, it did not go to plan. The idea is you create an app on your profile, then you receive a GitHub URL from Heroku to set up as a remote Repo. Then push the exiting code to this URL. Then it should be visible on your app’s website. This did not work. I received lots of errors. Without knowing the source of the error, I consulted goggle for answers. But none came. So, it was decided that there must be something fundamentally wrong with my program’s server even though everything worked perfectly in a development environment. I downloaded a Heroku template for NodeJS and adapted my program into this template. But still the error persisted. After some research, no answer came.

But then I discovered it in the build logs. These are logs of the build process Heroku provides for bug hunting. I had stored my API key in a variable for use in the development environment in a Js file. But this variable did not exist in a production environment and every time the build program was run it would be blocked looking for this environment variable. After fixing this, everything worked as planned, the panic receded and I was finally able to share the finished product with my teammates.

EXPLORING THE CONSOLE & HOW IT WORKS
      We signed up to Dialogflow and created a new agent at console.dialogflow.com.
      Enabled all Permissions asked by Google Cloud Console to let us have access to the Dialogflow console. We started playing around with the system, followed mainly guides on google Dialogflow
      Below is the interface of Dialogflow, that we learnt how to use from the official google page, it was important for us to fully understand all the functions within the module to maximise our final prototype, we are able to manage everything needed to create this chatbot.

The design of the Dialogflow Console (above) is extremely straight forward and clean which we absolutely loved, very easy to use. Here is the explanation of all the key elements to build out chatbot, something very important for us to understand in order to produce a fully functional prototype:

  • Agents : Our NLU (Natural Language Understanding) chatbot which we can give a name to
  • Intents : Here we use intents to categorize a user’s intentions, where it have training phrases, example might be the end-user ask our chatbot “Who are you?, “What is your name?” or “do you have a name?” these questions are different yet they have the same intension: to get the name of the Chatbot.
  • Entities : unlike intent entities used when end-user ask very specific information, like specific names, or addresses, or the number of units etc. The awesome thing about Dialogue flow is that there 3 types of entities, one of which is called system entities where it is already preloaded with numerous system entities, which allows our agent to extract existing information without having us to load them in ourselves, for example this would include like the current time or date, and location etc. The other two types of entities are called developer and session entities, which is when you need to develop your own entities, and entities that only apply to specific situations like booking a reservation for the end user.
  • Knowledge(Beta):This is something new that the V2 of Dialogflow introduced, where you can parse knowledge documents, for example FAQs, or Articles to find automated responses. The only limitation however, currently this is only available in English. This I think is extremely handy, since our chatbot is mainly currently FAQ based
  • Fulfillment: By default, our chatbot responds to a matched intent with a static response, however if you want to advance your chatbot, which is something we are hoping we would have time to get into, this is a “logic” or “smart” code that excutes when an intent is matched, our chatbot would then be able to perform an action or return a dynamic response by sending a webhook request to an external APIs and databases. This part is of cos more complicated, however if time allowed, we would give it a go.
